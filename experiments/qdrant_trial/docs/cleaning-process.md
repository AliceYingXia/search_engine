# Recipe JSON Cleaning Process

Reference for the cleaning step that prepares raw Workato recipe exports for semantic search ingestion.

Script: `clean_recipes.py`
Input: `example/*.json`
Output: `example/cleaned/<name>_semantic.json` and `example/cleaned/<name>_tracking.json`

---

## Why Two Output Files?

Each recipe produces two cleaned files because Qdrant stores two distinct things per point:

| File | Purpose | Becomes in Qdrant |
|---|---|---|
| `*_semantic.json` | Text content to embed | → vector |
| `*_tracking.json` | Structured metadata | → payload |

Splitting them at clean time keeps the concerns separate and makes the chunking step straightforward.

---

## The `as` Alias — Do Not Remove

Every codeline has an `as` field (e.g., `"bd6826bc"`). This is the **join key between steps** and must be preserved in both output files.

It serves two purposes:

1. **Data pill resolution** — when one step references another step's output, it uses `as` as the target identifier (e.g., `"line": "bd6826bc"` inside a `_dp()` pill). Without `as`, data flow between steps cannot be traced.

2. **Join key at chunking time** — the chunking script joins semantic and tracking files on `as` to build each Qdrant point (vector + payload). See [Identifying Tracking Info Per Chunk](#identifying-tracking-info-per-chunk) below.

### Synthetic `as` keys

When a codeline is missing `as` (a schema violation), a synthetic key `_step_{number}` is generated and stored in both files. This ensures the ancestry chain is never broken — children of the step can still resolve their `parent_as`. Synthetic keys are prefixed with `_` to distinguish them from real Workato aliases.

## What Is Kept vs. Stripped

### Per codeline — Semantic file

| Field | Kept | Notes |
|---|---|---|
| `as` | ✅ | Join key — must keep |
| `keyword` | ✅ | Step type |
| `provider` | ✅ | Which connector |
| `name` | ✅ | Operation name |
| `title` | ✅ | User-set display name |
| `description` | ❌ | Auto-generated by Workato UI — not updated when recipe logic changes; stale values would mislead embeddings |
| `comment` | ✅ | User-written documentation |
| `input` | ✅ | Field mappings — core content; includes `if` conditions |
| `uuid` | ❌ | Opaque identifier — not semantic |
| `number` | ❌ | Ordering — tracking concern |
| `extended_input_schema` | ❌ | Verbose API-fetched schema cache |
| `extended_output_schema` | ❌ | Verbose API-fetched schema cache |
| `dynamicPickListSelection` | ❌ | UI label cache |
| `toggleCfg` | ❌ | UI state |
| `visible_config_fields` | ❌ | UI state |
| `mask_data` | ❌ | Runtime flag |
| `unfinished` | ❌ | Build-time flag |

### Per codeline — Tracking file

| Field | Kept | Notes |
|---|---|---|
| `as` | ✅ | Join key |
| `uuid` | ✅ | Stable step identifier for deduplication |
| `number` | ✅ | Step ordering |
| `keyword` | ✅ | Step type |
| `provider` | ✅ | Connector — useful for Qdrant filtering |
| `name` | ✅ | Operation name |
| `parent_as` | ✅ | Added during cleaning — `as` of the parent block, or `null` for root |
| `depth` | ✅ | Added during cleaning — nesting depth (0 = root trigger); used for indentation in recipe summary |

### Recipe level — Tracking file

`flow.config` (connector list) is included at the recipe level in the tracking file. It is useful for filtering Qdrant results by connector.

## Recipe-Level Summary

Each semantic file includes a `recipe_summary` field — a static text block representing the full recipe. This is used as the recipe-level chunk for Qdrant (one point per recipe), separate from the step-level chunks.

It is generated from `provider` + `name` + `keyword` only — not from `description`, which can be stale. Nesting is shown via indentation derived from `depth` in the tracking file.

Example output:
```
Recipe: 2.2 Approved Sales Order updates SFDC and adds to KB
Connectors: workato_pub_sub, salesforce, workato_rag
Steps:
- trigger: workato_pub_sub / subscribe_to_topic
  - if
    - action: salesforce / create_custom_object
  - action: salesforce / update_sobject
  - action: workato_rag / upsert_knowledge
```

Note: `if` steps missing `provider` and `name` (a schema violation in the source data) appear with keyword only.

## `if` Conditions Are Preserved

The `input` field on an `if` codeline contains the full condition — operand type, logical operator, and each condition clause including any data pill references. Because `input` is kept in the semantic file, the `if` condition is fully preserved and available for embedding.

After pill resolution, the `lhs` value is also human-readable:
```json
{
  "keyword": "if",
  "input": {
    "type": "compound",
    "operand": "and",
    "conditions": [
      {
        "operand": "blank",
        "lhs": "[workato_pub_sub > message > sfdc_opportunity_id]"
      }
    ]
  }
}
```

### Complex `if` conditions

The script handles complex conditions correctly because `resolve_pills` is fully recursive — it walks any combination of nested dicts, lists, and strings inside `input`.

| Scenario | Handled | How |
|---|---|---|
| Multiple conditions in one `if` | ✅ | `resolve_pills` recurses into lists |
| Pills in both `lhs` and `rhs` | ✅ | Both are string values in a dict — both resolved |
| Nested compound groups (AND of ORs) | ✅ | Dict recursion goes to any depth |
| `if` nested inside another `if`'s block | ✅ | `walk_codelines` recurses into `block` arrays |
| `foreach` or `try` containing an `if` | ✅ | Any block nesting depth is handled |
| Formula values (e.g. `=now + 30.days`) | ✅ kept as-is | Not a pill — regex does not match; formula string is semantically meaningful |

### Known noise: condition-level `uuid`

Each condition clause contains its own `uuid` (e.g., `"uuid": "80594e80-..."`). This is a Workato internal identifier that sits inside `input` and is currently preserved in the semantic file. It is skipped during `input` flattening in the chunking step, so it does not appear in embed text.

## Data Pill Resolution

Raw data pills in `input` values are replaced with human-readable labels during cleaning. This applies only to the semantic file — the tracking file retains raw pills.

### Raw format

```
#{_dp('{"pill_type":"output","provider":"workato_api_platform","line":"bd6826bc","path":["context","calling_ip"]}')}
```

### Resolved format

```
[workato_api_platform > context > calling_ip]
```

### Pill formats

Two `_dp()` formats exist in Workato recipes and both are resolved:

| Format | Example | When used |
|---|---|---|
| `#{_dp('...')}` | `"Name": "#{_dp('...')}"` | Standard interpolation inside a string |
| `=_dp('...')` | `"file_content": "=_dp('...')"` | Formula mode — entire value is a pill expression |

### Resolution rules

| `pill_type` | Resolved label |
|---|---|
| `output` | `[{provider} > {path[0]} > {path[1]} > ...]` |
| `job_context` | `[job_context > {path[0]} > ...]` |
| other / unknown | `[{pill_type} > {path[0]} > ...]` |

`job_context` pills reference Workato runtime metadata (e.g., `job_created_at`) rather than another step's output — they have no `provider` or `line`, so the type itself is used as the prefix.

When `path` is empty (e.g. `"path": []`), the label is just `[{provider}]` — meaning the whole output object of that step is referenced, not a specific field.

### Examples from real recipes

| Raw pill | Resolved |
|---|---|
| `output`, provider `workato_api_platform`, path `[context, calling_ip]` | `[workato_api_platform > context > calling_ip]` |
| `output`, provider `workato_pub_sub`, path `[message, order_number]` | `[workato_pub_sub > message > order_number]` |
| `job_context`, path `[job_created_at]` | `[job_context > job_created_at]` |

Resolution is applied recursively to all string values inside `input`, including nested objects, arrays, and condition clauses in `if` steps.

## No LLM Required

The cleaning step is fully deterministic. Static Python code (`clean_recipes.py`) handles all transformations:
- Field extraction and removal
- Recursive traversal of `block` arrays
- Data pill resolution in `input` values
- `parent_as` and `depth` derivation
- Recipe-level summary generation

No language model is needed at this stage.

## Identifying Tracking Info Per Chunk

After chunking, every chunk retains the `as` field from the semantic file. This is used to look up the matching entry in the tracking file and attach it as Qdrant payload:

```
semantic file          tracking file
  step["as"] ────────── step["as"]
       │                     │
  embed text            attach as payload
       │                     │
       └──── Qdrant point ───┘
             vector + payload
```

Each Qdrant point ends up with:
- **vector**: embedding of the semantic content
- **payload**: `{ as, uuid, number, parent_as, keyword, provider, name, recipe_name }`

### Edge case: missing `as`

If a codeline is missing `as` (a schema violation in the source data), both the semantic and tracking entries will have `as: null`. The chunking script should fall back to matching on `number` in this case.
